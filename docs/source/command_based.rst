Command-Based 程式設計
======================

WPILib 支援一種常見的 FRC 程式設計方式，稱為 Command-Based 程式設計。這不是唯一的寫法，但它非常適合管理複雜機器人。

Command-Based 是什麼？
----------------------

Command-Based 是一種程式設計模式，讓你把機器人行為拆成兩種角色：

- Subsystems：代表特定硬體區塊，如驅動底盤、射球機構、傳送機
- Commands：代表機器人要執行的動作，如「前進」、「轉向」、「發射」

這種方式可以讓程式更清楚、易讀，而且不同功能彼此之間不會互相干擾。

為什麼 Command-Based 有用？
---------------------------

傳統寫法在 `teleopPeriodic()` 中每次都檢查按鈕、讀取感測器、下馬達指令。當機器人越來越複雜時，這樣的程式會變得難維護。

Command-Based 的好處：

- 增加程式可讀性
- 易於重複使用相同動作
- 可以讓系統自動管理哪些命令可以同時執行
- 更容易把硬體細節封裝起來

Subsystems 與 Commands
-----------------------

- Subsystem：負責控制硬體，像是馬達、編碼器、氣動閥。它代表一個整體功能區域。
- Command：使用 Subsystem 做某件事，像是「如果按下按鈕，就啟動射球機構」。

簡單比喻：Subsystem 是汽車，Command 是按油門、剎車或方向盤的操作。

Command-Based 的行為描述
-------------------------

Command-Based 偏向「宣告式」程式設計。你不是每次都寫「如果按鈕按下，就做什麼」，而是先定義好命令，然後告訴系統「當按鈕按下時執行這個命令」。

這讓程式更簡潔，例如：

- 定義一個 `DriveForwardCommand`
- 定義一個 `DriveSubsystem`
- 當司機按下前進時，系統自動執行 `DriveForwardCommand`

若要改用其他硬體或不同機構，只要改 `Subsystem` 的內部實作，命令本身通常不需要改動。

何時應該學 Command-Based？
---------------------------

對於新手來說，剛開始可以先用 `TimedRobot` 寫簡單程式，一旦機器人功能變多、按鈕越來越多，就可以考慮改用 Command-Based。它適合：

- 多個子系統互相協作的機器人
- 想讓程式更容易維護的隊伍
- 想重用去年程式或多機器人專案的情況

結論
----

Command-Based 不是為了讓你一開始就複雜化程式，而是為了讓你的機器人程式在成長時仍然能保持整齊。

如果你剛開始，先試試 `TimedRobot`；當你熟悉 WPILib 後，再回來學 Command-Based，會更容易理解。
